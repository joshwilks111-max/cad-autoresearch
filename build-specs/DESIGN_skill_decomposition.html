Should we break the worker into skills?

An interactive walk through whether decomposing the single worker prompt into specialized skills will lift the score ceiling on hard CAD parts — and what to actually build first.

The bottom line

The skill-based design the intuition reached for already exists in the repo — but it was never plugged into the training loop, so every score on the board came from the old single-prompt worker. The hypothesis has not actually been tested yet.

The honest answer is a scoped yes: the ceiling on hard parts splits into a part that no prompt change can move (a limit baked into the grader) and a part that better instructions genuinely can move (reading drawings, not missing features, not crashing the geometry kernel). Decomposition can only help the second part — so build the cheap version first and let evidence decide whether the elaborate version is ever worth writing.

1 · What's actually being asked

The worker is an AI agent that writes a small program to rebuild a hidden target part. It gets a score, a hint about what's wrong, and tries again — a tight propose → score → revise looprun_inner_loop.py — the worker calls a proposer, the harness builds + grades, feedback is fed to the next turn. ≥0.95 composite = "solved". you'll recognise from RL.

Today that worker follows one big instruction file (the monolithic promptprogram.md (213 lines). The candidate worker is spawned as one `claude -p` call told "Read ./program.md and follow it exactly."). The intuition under test: a generalist following one broad manual hits a generalist's ceiling — so break it into specialists, each expert at one kind of difficulty, with a smart coordinator picking which one to use.

The reframe that changes everything: a first cut of exactly this — three skill packsskills/drawing-read, skills/cad-reconstruct, skills/cad-grade + skills/RESOLVER.md (the dispatcher table). Each has a SKILL.md + evals/. plus a dispatcher — already sits in the repo. But a quick check of the loop code shows it is never loaded by the running worker (the worker is only ever told to read the one big manual). So the 13-of-13 solves and the floored hard parts all came from the generalist. the experiment hasn't run yet

There's also a subtle mismatch. The skills that exist are split by pipeline stage (read the drawing → model it → score it). The intuition is about splitting by axis of difficulty (round vs prismatic, few faces vs many, one body vs many). Those are different cuts. And tellingly, the hardest stage — reading the drawing — is already its own skill and is still stuck at the field-wide accuracy wall. So isolating a stage didn't lift that ceiling, because that ceiling is about model eyesight, not organisation.

2 · The ceiling is two different numbers

"Hard parts cap around 0.8" hides the real story. The scores split into two regimes, and decomposition only has room to help in one of them.

Every part on the board, by score

solved (spec given)  ·  floored but near its grader cap  ·  floored far below its cap (real room to improve)  ·  the 0.95 "solved" line
Spec track: 13 / 13 solved. When the dimensions are handed over, the generalist already nails it — the scoring gradient is smooth and steep near the top. No room for decomposition to help here.
Drawing / hard track: floored. But look where. Some parts sit just under a hard grader cap (little room). Others — the ~0.17 and ~0.24 ones — are just empty boxes that were never actually rebuilt: huge room, untouched by the generalist.
The key distinction. "The ceiling" conflates the grader cap (≈0.88 for a busy part — unmovable by any instruction) with the achieved score (0.17–0.69). The gap between them is the only room decomposition can win. On the empty-box parts that gap is ~0.7 (large). On the spec parts it's zero (already at the top).

4 · Does decomposition lift the ceiling? Both sides, hard.

The owner asked for the truth, not validation — so here is the strongest version of each case, built by two adversarial reviewers. Toggle to read one side, the other, or both.

✕ The skeptic: it reorganises the same failure

  • Three of the five floored parts are at their grader cap. No program a better-organised agent could write scores materially higher — the math forbids it.
  • The only score the loop actually optimises is the single composite number. Add many skills and you add many chances for a negative-transfer skillSkillLens (microsoft.github.io/SkillLens): ~25% of skill transfers are net-negative; utility is NOT predicted by textual plausibility — worse than chance. you can't even detect, because one number can't tell you which skill helped.
  • The hard parts are hard because of what they are (busy, multi-body) and what you can see (40–77% of the dimensions). A cheaper specialist model reads fewer dimensions and reasons less — weaker on the one thing that's hard.
  • Higher-leverage instead: fix the round-part glitch, get a better drawing-reading tool, add simple real parts. None of those is "reorganise the prompt".

✓ The believer: attribution makes it climbable

  • The generalist leaves measurable room on the table: a 77%-accurate drawing reader sits unused behind the worker's own ~40% reading; the empty-box parts were never actually rebuilt.
  • Each axis owns a different, clean signal — and the scorer was already shaped so that on busy parts the volume-overlap scoreIoU (volumetric). reward.py shifts weight INTO it for feature-rich parts via adaptive_feature_weighting, keyed on the hidden answer's face count so it can't be gamed. is the sensitive one. So you gate each skill on its signal, not the flat total.
  • Where a skill can't use the total (it's capped), it gates on an off-total signal — drawing-reading accuracy, missing-material, build-success — exactly the discipline the literature says works.
  • Honest scope: this wins on the drawing parts and the empty-box parts. It does not move the capped parts. (The believer concedes the skeptic owns that ground.)
What actually decides it: both cases collapse to one testable question — does splitting the work and routing beat just iterating with the current worker, at equal cost? That's answerable cheaply (the ladder below). Until it's answered, the honest move is: decompose narrowly, prove the routing earns its complexity, and be ready to stop early.

The decisive finding from the review: the worker already gets a per-axis "what's wrong" hint every turn (the feedback already names the four failure types and is already injected into the next prompt). So Rung 1.5 — a ~10-line smarter prompt — probably captures most of the value with no router at all. The full router only earns its keep if a fixed/learned chooser beats a smart agent choosing for itself. It may never be worth building.

6 · The hard part: which skill gets the credit?

With many skills, how do you know which one moved the score? A single total can't say. This is what makes or breaks a per-skill loop — and the substrate is mostly already there.

Each axis owns a different signal

The scorer already returns a detailed breakdown (not just the total) and it's already saved on every attempt. The fix isn't new plumbing — it's a small readerA loop-local reader over breakdown.raw (already persisted in every ledger row). The earlier 'persist raw' framing was wrong — the data is already on disk; only a reader is missing. that lays the signals out side by side.

reading the drawing
dimension-recall accuracy — off-total, measured before any building (and only offline, not live in the loop)
missing features
volume-overlap + volume error — the only layers that drop with missing material
geometry crashes
did a solid build at all? + how much shrank — independent of everything else
wrong strategy
overlap stuck while volume is fixable; the seam-vs-missing split in the face count
The honest limit: on a capped part the total is flat — so the skill there must gate on an off-total signal (reading accuracy, missing-material, build success), never the total. That's exactly the "gate on the real failure mechanism, not on how plausible it sounds" rule from the skill-research literature.

7 · What the review gauntlet caught

A four-lens review (strategy · architecture · developer-experience, two independent reviewers each) was run over this design. It found real problems — including a few where the first draft was simply wrong. Every one is now fixed. Expand any card.

The pattern in what got caught: the first draft kept overstating how clean the idea was — claiming zero code changes when one was needed, claiming a signal was live when it was offline-only, claiming a data path that didn't exist. The strategy and architecture reviewers independently reached the same verdict: the idea is sound, the architecture is over-built — build the cheap rungs and let evidence decide the rest. The core thesis survived; the engineering honesty got fixed.

8 · Verdict & the seven questions

1 · What are the axes?
Four worker-fixable (reading, completeness, robustness, strategy) + three grader-walls (face-count, hollow voids, round-part noise). §3.
2 · Does it lift the ceiling?
Yes on the worker-fixable parts; no on the capped ones. The gap from achieved-score to grader-cap is the prize. §2, §4.
3 · The architecture
A routing mode of the existing worker-spawner — reads the per-axis breakdown, picks a skill prompt + model. Cheap models for skills, smart model as chooser. §5.
4 · Credit assignment
Each axis owns its own signal; the breakdown is already saved; a small reader exposes it. Capped axes gate off-total. §6.
5 · Search philosophy
Parallel cheap probes + a chooser, not assumed steady progress. "100 shots; take the one that hits." A learned chooser only after fixed rules win. §5 rungs 2–4.
6 · Is the gate higher?
Yes — the real metric is "did it rebuild a part start-to-finish with no human," measured as turns-to-solve on a held-out part. Not "did a tweak nudge a number."
7 · Where does vision fit?
A tool the worker can reach for, never a forced step. Point it at the 77% reader; let the loop learn when to use it. (But it's a metered, networked call — gate and cache it.)
Recommendation. Scoped yes, resized down by the review. Run the parts-and-glitch track first (it lifts what decomposition can't, and feeds it). Build the measurement step, the one-line tool pointer, and the ~10-line smarter prompt — and let that evidence decide whether the full router ever gets written. Don't build a topology skill. Don't build the learned chooser until fixed rules have already won.
